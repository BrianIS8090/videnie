# AGENTS.md

Этот файл задает рабочие правила для агентных кодеров в репозитории `videnie`.
Цель: вносить изменения в стиле текущего проекта, не ломая сборку и UX.

## 1) Контекст репозитория

- Стек: React 19 + TypeScript + Vite 7 + Tailwind CSS + Framer Motion.
- Пакетный менеджер: `npm` (есть `package-lock.json`).
- Линтинг: ESLint Flat Config (`eslint.config.js`).
- TypeScript: strict-mode включен (`strict: true`).
- Хранение данных: IndexedDB (`src/utils/imageDB.ts`).
- Основная UI-логика сейчас сосредоточена в `src/App.tsx`.

## 2) Быстрый старт для агентов

1. Установить зависимости: `npm install`
2. Локальная разработка: `npm run dev`
3. Проверка линтера: `npm run lint`
4. Проверка сборки: `npm run build`
5. Локальный предпросмотр сборки: `npm run preview`

## 3) Build / Lint / Test команды

### Актуальные npm scripts (из package.json)

- `npm run dev` — старт Vite dev-сервера.
- `npm run build` — `tsc -b && vite build`.
- `npm run lint` — `eslint .`.
- `npm run preview` — просмотр production-сборки.

### Важные детали

- `build` уже включает TypeScript compile check через `tsc -b`.
- Отдельного `typecheck`-скрипта пока нет.
- Тестовый раннер в проекте сейчас не настроен.

### Что с тестами сейчас

- В репозитории нет test/spec файлов.
- В `package.json` нет скрипта `test`.
- Команда вида `npm test` сейчас не является источником истины для проекта.

### Как запускать один тест (когда тесты будут добавлены)

Используйте фактический test-скрипт из `package.json` (проверка: `npm run`).

Примеры для типового сценария с Vitest:

- Один файл: `npm run test -- src/path/to/file.test.ts`
- Один кейс по имени: `npm run test -- -t "имя теста"`
- Файл + имя теста: `npm run test -- src/path/to/file.test.ts -t "имя теста"`

Примеры для Jest (если проект перейдет на Jest):

- Один файл: `npm run test -- src/path/to/file.test.ts`
- Один кейс: `npm run test -- -t "имя теста"`

Примечание: не добавляйте команды в документацию, пока они не закреплены в `scripts`.

## 4) Структура проекта (ориентиры)

- `src/main.tsx` — входная точка приложения.
- `src/App.tsx` — основной экран и orchestration UI/данных.
- `src/utils/imageDB.ts` — IndexedDB-операции (история изображений, стоимость).
- `src/lib/utils.ts` — общие утилиты (`cn` для Tailwind-классов).
- `src/index.css` — глобальные стили, токены и анимации.
- `tailwind.config.ts` — дизайн-токены, анимации, тема.

## 5) Правила стиля кода (по наблюдаемым паттернам)

### Форматирование

- Отступ: 2 пробела.
- В коде без точек с запятой.
- В `src/*.ts(x)` в основном используются одинарные кавычки.
- Допускается формат, который уже принят в конкретном файле (не делать шумных перестановок).

### Импорты

- Порядок импорта держать стабильным:
  1) внешние пакеты,
  2) внутренние модули,
  3) стили (`.css`).
- Есть alias `@/*` в `tsconfig.json` и `vite.config.ts`; можно использовать для новых модулей.
- Не оставлять неиспользуемые импорты (`noUnusedLocals: true`).

### TypeScript и типы

- `strict: true` обязателен для новых изменений.
- Предпочитать `interface`/конкретные типы вместо неявных структур.
- Избегать `any`; если требуется гибкость — сначала пробовать `unknown` + narrowing.
- Учитывать `noUnusedLocals`, `noUnusedParameters`, `noFallthroughCasesInSwitch`.
- Для nullable-полей в проекте используется явный `| null`.

### React-подход

- Функциональные компоненты + hooks.
- Состояние через `useState`, эффекты через `useEffect`, ссылки через `useRef`.
- Обработчики событий именовать в стиле `handleXxx`.
- Для коллекций использовать стабильные `key`.
- Не выносить в рефакторинг архитектуру компонента без явной задачи.

### Нейминг

- Компоненты/типы: `PascalCase` (`GeneratedImage`, `SourceImage`).
- Переменные/функции: `camelCase` (`loadInitialImages`, `handleDeleteImage`).
- Константы верхнего уровня: `UPPER_SNAKE_CASE` (`MODELS`, `ASPECT_RATIOS`).
- Названия должны отражать домен генерации изображений, а не абстрактные `data1`/`temp`.

### Error handling и логирование

- Асинхронные операции оборачивать в `try/catch`.
- Ошибки логируются через `console.error(...)` с контекстом.
- Для пользовательских ошибок в UI используются `alert(...)`.
- Не глотать ошибки пустыми `catch`.

### Работа с IndexedDB

- Все операции БД идут через `src/utils/imageDB.ts`.
- Не дублировать прямые обращения к IndexedDB в новых местах без причины.
- При добавлении новых полей синхронно обновлять тип `StoredImage` и потребителей.

### UI и Tailwind

- Используется utility-first подход и токены через CSS variables.
- Для объединения классов применять `cn()` из `src/lib/utils.ts` при необходимости.
- Сохранять существующую glassmorphism-стилистику и анимации проекта.

## 6) Линт и качество изменений

Перед завершением любой задачи агент должен:

1. Запустить `npm run lint`.
2. Запустить `npm run build`.
3. Если добавлены тесты — запустить релевантный тестовый скрипт.
4. Убедиться, что не появились лишние зависимости и неиспользуемый код.
5. Если есть падения, явно отделять новые проблемы от уже существующих в проекте.

## 7) Правила по изменениям файлов

- Делать минимальные, локальные правки по задаче.
- Не переименовывать массово сущности без необходимости.
- Не смешивать функциональные изменения с косметическим рефакторингом.
- Новые файлы размещать рядом с соответствующим модулем.
